Keyboard Navigation Rework — 3 Approaches

How to turn jot into a keyboard-first app that still works with mouse

A

Unified Focus Engine (Recommended)

Build a single useFocusEngine hook that manages a focus tree — panes → regions → items. Every focusable thing in the app registers itself. One state machine controls modes (NORMAL / INSERT / COMMAND). One place handles all key dispatch.

Pros

  • Single source of truth for focus & mode state
  • h/j/k/l works everywhere consistently
  • Escape always does the right thing (drill up the focus tree)
  • New components get keyboard nav "for free" by registering
  • Mode indicator is trivial — just read the engine state
  • Mouse clicks just update the same focus state

Cons

  • Biggest rewrite — replaces all existing keyboard handling
  • Need to carefully migrate each component
B

Layered Keymap System

Keep existing per-component handlers but layer them through a central keymap registry. Each component pushes/pops its keymap onto a stack. Top of stack gets first crack at keys. Like tmux's key-table system.

Pros

  • Less invasive — components keep their handlers
  • Keymap stack naturally handles modals/dialogs
  • Can migrate incrementally

Cons

  • Doesn't fix the focus model — still no unified "where am I"
  • Components still manage their own focus state
  • Mode indicator needs separate plumbing
  • Harder to make h/l column nav work consistently
C

Fix What's Broken

Keep the current architecture but fix the specific issues: wire up h/l in kanban, make task editor fields Tab-navigable, add mode indicator, fix Escape behavior. Targeted patches.

Pros

  • Smallest change — less risk
  • Fast to implement

Cons

  • Doesn't fix the root cause — scattered keyboard handling
  • Will keep feeling "browser-y" not "TUI-y"
  • Each new feature needs its own keyboard plumbing
  • Will accumulate more bugs over time
